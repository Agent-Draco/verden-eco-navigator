### Aim of the change:
The primary goal was to fix a critical rendering bug in the CesiumJS-based navigation system where the 3D vehicle model would become distorted, fragmented, or disappear entirely during camera rotation. The fix involved enforcing a strict separation between dynamic, velocity-driven entity orientation and static, corrective model transformations.

### What was changed:
- **Removed Manual Entity Orientation**: All code that directly assigned an orientation to the vehicle entity using `Cesium.Transforms.headingPitchRollQuaternion` was removed from `src/components/verden/VehicleEntityController.ts`.
- **Enforced Velocity-Based Orientation**: The entity's orientation is now exclusively driven by `new Cesium.VelocityOrientationProperty(positionProperty)`, ensuring its rotation is derived purely from its movement direction.
- **Applied Static Model Correction**: A `nodeTransformations` property was added to the entity's model definition. This applies a static `Cesium.Quaternion.fromHeadingPitchRoll` to the model's root node, correcting its intrinsic alignment (e.g., from Blender's Y-up, Z-forward to Cesium's Z-up, X-forward) without interfering with the dynamic entity orientation.

### Result of the change:
The vehicle model now renders smoothly and correctly, remaining stable and properly oriented regardless of camera movement or rotation. The conflicting transforms were eliminated, resulting in a stable and predictable rendering pipeline as specified by the CesiumJS best practices.

### Detailing of what you've understood overall:
I understood that CesiumJS has a strict rendering pipeline where dynamic and static transformations must be kept separate. An entity's orientation should reflect its movement in the world (dynamic), while a model's `nodeTransformations` should only be used to correct its internal, built-in alignment (static). Attempting to mix these by manually rotating the entity while also using a velocity-based orientation property leads to undefined behavior and visual artifacts. The correct solution was to remove the manual entity-level rotation and apply a one-time, static correction directly to the model's internal hierarchy, which is exactly what was implemented.
