
**Aim of the change:**
To resolve the "Maximum update depth exceeded" error that occurs on the home page, which causes the application to crash.

**What was changed:**
The `useEffect` hook in `src/pages/Home.tsx` responsible for syncing the navigation bar's visibility was identified as the source of an infinite re-render loop. The `setNavHidden` function, passed from `useApp` context, was included in the dependency array. Since context provider functions can be redefined on each render, this caused the effect to run continuously. The fix involved removing `setNavHidden` from the dependency array.

**What was the result of the change:**
The infinite loop is eliminated. The home page now renders correctly without crashing, and the navigation bar visibility is still correctly synced with the search overlay's state.

**What you inspected to determine course of action:**
I inspected the browser's JavaScript console logs, which showed a "Maximum update depth exceeded" React error originating from the `Home` component. I then analyzed the source code of `src/pages/Home.tsx`, specifically looking at the `useEffect` hooks and their dependencies. The hook at line 54 was the only one that could be causing this issue, as its dependency `setNavHidden` is a function coming from a context, which is a common pattern for this type of bug.

**Tests done on the app after the change:**
1.  Loaded the home page and verified that the "Maximum update depth exceeded" error no longer occurs.
2.  Clicked on the search bar to open the search overlay and confirmed the navigation bar disappears.
3.  Closed the search overlay and confirmed the navigation bar reappears.
4.  Navigated to other pages and back to the home page to ensure the fix did not introduce any regressions.

**Detailing of what you've understood overall:**
I understand that React's `useEffect` hook re-runs whenever any of its dependencies change. When a dependency is an object or a function (non-primitive), its reference can change on every render even if its content is the same, leading to infinite loops if it triggers a state update within the effect. It's crucial to either memoize such functions using `useCallback` in the provider or omit them from dependency arrays if they are stable and don't need to trigger the effect. In this case, omitting the dependency was the simplest and safest solution.
