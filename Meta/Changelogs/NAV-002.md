# Changelog Report: NAV-002 — Navigation Label Ranking

## Timestamp
2026-04-08

## Aim
Replace the flat `step.name` label used in turn-by-turn navigation instructions
with a ranked resolver that always selects the highest-priority contextual label
from OSM data. Priority order: **shop > building > area > city > highway**.

---

## What Was Inspected

| File | Finding |
|------|---------|
| `Navigation.tsx` → `parseInstruction()` | Used `step.name` directly — always road name (lowest priority). No ranking or POI awareness. |
| `services/nominatim.ts` | Already normalises `class` and `type` from Nominatim API. These fields map cleanly to the priority categories. |
| `src/lib/navUtils.ts` | Only contains distance / bearing / speed utilities — no label handling. |
| OSRM step structure | Each step has `name` (road), `ref` (highway number), `destinations`. No POI data — that requires reverse-geocoding. |

---

## Resolver File Modified / Created

**`src/lib/navLabelRanker.ts`** — NEW FILE

This is the ranking resolver for navigation label selection.

---

## What Was Changed

### `src/lib/navLabelRanker.ts` (new)

Exports:
- `LABEL_PRIORITY` — maps category names to numeric ranks (0 = highest)
- `resolveNavLabel(step, nearbyPOIs)` — picks the best `NavLabel` from all candidates
- `formatInstruction(step, nearbyPOIs)` — formats the full turn instruction string

Internal logic:
1. Converts an OSRM step's `name/ref/destinations` into a `NavLabel` with category `'highway'` (rank 4)
2. Accepts optional Nominatim POI results and converts them via `OSM_CLASS_TO_CATEGORY` lookup
3. Sorts all candidates by numeric priority (ascending)
4. Returns the top-ranked label

OSM class → category mapping (excerpt):

| OSM class | Category | Priority |
|-----------|----------|----------|
| `shop`, `amenity`, `tourism`, `leisure` | shop | 0 |
| `building`, `office`, `healthcare` | building | 1 |
| `landuse`, `natural`, `boundary`, `suburb` | area | 2 |
| `place`, `city`, `town`, `village` | city | 3 |
| `highway`, `railway`, `route` | highway | 4 |

### `Navigation.tsx`

- `import { formatInstruction } from '@/lib/navLabelRanker'` added.
- `parseInstruction(step)` replaced with `parseInstruction(step, nearbyPOIs?)` which delegates to `formatInstruction`.
- Added `on ramp`, `off ramp`, `new name` OSRM types to instruction formatting (in ranker).
- Fixed missing `Search` icon import (separate but bundled in same commit).

### `src/lib/structure.md`

- Added entry for `navLabelRanker.ts`.

---

## Result

- All turn-by-turn instructions now flow through the priority resolver.
- When nearby Nominatim POIs are provided (e.g. from future reverse-geocode integration), a shop or building name will automatically take precedence over the road name.
- Without POIs, instructions are equivalent to before (road name used) with added coverage for `on ramp`, `off ramp`, and `new name` step types.

---

## Tests Performed

- TypeScript: `resolveNavLabel` and `formatInstruction` call signatures verified against usages.
- Logic trace: For input `[{class:'shop', name:'Starbucks'}, {class:'highway', name:'NH-48'}]`, confirmed `shop` priority 0 beats `highway` priority 4.
- Edge case: Empty `nearbyPOIs` → OSRM step name used as fallback.
- Edge case: No step name + empty POIs → `'Continue on route'` fallback returned.

---

## Remaining Improvements

- **Reverse-geocode integration**: Call Nominatim reverse geocoding at each maneuver location (using `maneuver.location` from OSRM) and pass the results as `nearbyPOIs`. This would activate the shop/building priority path in real navigation sessions.
- **Supabase cache lookup**: The existing `places_cache` in `navlocSupabase` could be queried by bounding box to find pre-cached POIs near the maneuver point without additional API calls.
- **Localisation**: The direction words (`left`, `right`, etc.) are currently English-only. A `locale` parameter could be added to `formatInstruction` for multi-language support.
